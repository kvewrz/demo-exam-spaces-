# Toast: простая и удобная схема уведомлений

## Зачем toast

Нужен **ненавязчивый** способ сообщить об успехе или ошибке без перезагрузки страницы: сохранён профиль, создана бронь, сервер вернул ошибку. Alert-браузера для этого не подходит (блокирует поток, выглядит чужеродно).

## Идея архитектуры

1. **Один контейнер** в корне приложения (часто в портале к `document.body`), который рисует стек уведомлений в углу экрана.
2. **Функция уровня приложения** — условно `notify({ type: 'success' | 'error' | 'info', message: string })`, доступная из любого места через контекст или лёгкий store.
3. Каждое уведомление живёт несколько секунд и **исчезает** само; для ошибок можно сделать дольше или с кнопкой «Закрыть».

Типы:

- **success** — зелёный/нейтральный акцент: создание, сохранение.
- **error** — красный/акцент ошибки: 4xx/5xx, сеть недоступна.
- **info** — нейтральные подсказки (редирект, «скопировано»).

## Привязка к запросам

### TanStack Query

В `useMutation` используйте **`onSuccess`** и **`onError`**:

- `onSuccess` → `notify({ type: 'success', message: '…' })`.
- `onError` → разбор `error`: если это `AxiosError`, взять `response?.data?.error` или общий текст.

Так toast остаётся рядом с бизнес-действием и не размазывается по компонентам случайными `alert`.

### RTK Query

В **`unwrap()`** после `await dispatch(api.endpoints.xxx.initiate(...)).unwrap()` обрабатывайте успех; в `catch` показывайте ошибку. Либо используйте патч через **`extraReducers`** для глобальных эффектов (осторожно — не дублируйте сообщения).

## Подходы: библиотека vs своё

| Подход | Плюсы | Минусы |
|--------|--------|--------|
| **Готовая библиотека** (react-hot-toast, sonner, notistack) | Быстро, анимации, очередь | Зависимость, стилизация |
| **Свой контекст + массив сообщений** | Полный контроль, минимум кода | Нужно продумать z-index и a11y |

Для учебного проекта достаточно **одной** выбранной схемы и последовательного вызова из мутаций.

## Итог

Toast — это **тонкий слой** над результатом запроса: одна точка входа `notify`, типы сообщений и связка с `onSuccess`/`onError` мутаций дают предсказуемый UX без дублирования логики в каждой форме.
