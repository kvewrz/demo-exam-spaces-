# Guard и роли: концепция (без кода)

## Зачем нужен guard

**Защищённый маршрут** — это страница или действие, которые имеют смысл только для авторизованного пользователя или только для определённой **роли**. Guard отвечает на три вопроса:

1. Пользователь **вошёл** в систему (есть действующий токен или сессия)?
2. Его **роль** допускает это действие?
3. Что показать, если доступа **нет**?

## Три разных «нет доступа»

| Ситуация | Типичный ответ сервера | Поведение клиента |
|----------|------------------------|-------------------|
| Нет токена или он не передан | **401 Unauthorized** | Редирект на страницу входа; при необходимости сохранить «куда вернуться». |
| Токен есть, но истёк или подпись неверна | **401** | Очистить устаревшие данные, попробовать refresh (если используете), иначе — снова вход. |
| Токен валиден, но роль не та | **403 Forbidden** | Не редиректить без объяснения: сообщение «недостаточно прав», ссылка «на главную». Для API то же — показать toast и не ломать весь экран. |

Важно **не путать** 401 и 403 с «ошибкой формы»: это именно **авторизация и права**, а не валидация полей.

## Два уровня защиты

1. **UI (меню и ссылки)** — скрываете пункты, которые пользователю всё равно недоступны. Это удобство, не безопасность: пользователь может ввести URL вручную.
2. **Маршрутизатор** — при заходе на `/manage-bookings` без роли manager роутер показывает fallback (редирект, страница «нет доступа»).
3. **Сервер** — источник истины: даже при ошибке на фронте API вернёт 403. Клиент обязан корректно отобразить ответ.

## Где хранить «кто я» после логина

Обычно сохраняют **access token** (и при необходимости refresh) в `localStorage` или памяти + **профиль пользователя** (id, email, role), чтобы не дергать `/users/me` на каждый клик. После логина имеет смысл один раз загрузить профиль и положить в контекст / Redux / React Query cache.

Роль из профиля используйте для **условного рендера** меню и для **выбора дочерних маршрутов** (вложенные `<Route>` с проверкой роли).

## Итог

Guard — это согласованная политика: **токен**, **роль**, **маршруты**, **обработка 401/403** и понятный UX. Реализация на React (Outlet, loader, компонент-обёртка) — отдельный шаг; здесь важно понимание модели.
